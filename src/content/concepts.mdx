# Local Storage

The SDK stores all conversation data locally in the browser using WatermelonDB,
an offline-first database built on top of IndexedDB.

## Why Local Storage?

The Portal API is stateless — it processes your request and returns a response,
but doesn't remember anything. This keeps the API simple and privacy-friendly,
but means your app needs to handle persistence.

The SDK solves this by automatically saving messages to a local database. When
you call `sendMessage`, the SDK:

1. Saves the user message to the local database
2. Sends the request to the Portal API
3. Streams the response back to your app
4. Saves the assistant message to the local database

All of this happens automatically when you use `useChatStorage`.

## What Gets Stored

Conversations contain metadata:

- Unique ID
- Title
- Timestamps (created, updated)
- Optional project ID for grouping

Messages contain:

- Role (user, assistant, system)
- Content (text)
- Model used for the response
- File attachments (metadata only — actual files stored separately)
- Embedding vector (for semantic search)
- Token usage stats

## Offline Support

Because data lives locally, your app works offline. Users can browse previous
conversations without a network connection. New messages queue up and sync when
connectivity returns.

## Data Ownership

All data stays on the user's device. The Portal API never stores conversation
history. Combined with optional wallet-based encryption, users have full control
over their data.
