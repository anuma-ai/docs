# Files and Images

Messages can include files in two directions: attachments you send to the model,
and content the model generates. These work differently.

## Attachments

Attachments are files users share with the AI — documents, images, spreadsheets,
archives. The SDK processes these before sending to make them useful as context.

### Document Processing

For documents, the SDK extracts text content so the model can understand what's
in the file. Raw binary files aren't useful to language models — they need text.

The SDK includes processors for common document types:

- **PDF** — Extracts text content from all pages
- **Word (.docx)** — Extracts raw text without formatting
- **Excel (.xlsx, .xls)** — Converts spreadsheets to structured JSON, preserving
  sheet names and data organization
- **ZIP archives** — Extracts and processes nested files recursively, returning
  a structured summary with content from each supported file inside

Processors run automatically when you send a message with file attachments. The
extracted text is included as context for the model. File metadata is preserved
separately so your app can display what files were attached.

You can extend the processor system with custom processors for additional file
types.

### Images

Images are sent directly to the model without text extraction. Vision models
analyze the visual content — identifying objects, reading text in images,
understanding charts, or answering questions about what they see.

Multiple images can be included in a single message. The model sees all of them
and can compare or reference specific images in its response.

For images that contain text you want extracted, OCR is available as a separate
utility.

## Generated Content

When the model generates files (like images from the image generation tool),
the SDK handles downloading and storing them locally.

Generated image URLs from the API are temporary. The SDK downloads images
automatically and stores them encrypted on the user's device. This ensures
images remain available in conversation history even after the original URLs
expire.

The stored files are encrypted at rest and tied to the user's session. When
displaying messages, the SDK resolves stored files to displayable URLs
on-demand.
